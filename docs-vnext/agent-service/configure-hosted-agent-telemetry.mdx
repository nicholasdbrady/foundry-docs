title: "Export hosted agent telemetry using OpenTelemetry"
description: "Export traces, logs, and metrics from a Foundry hosted agent to Application Insights or any OpenTelemetry (OTLP) endpoint."
---

Configure a [hosted agent](/agent-service/hosted-agents) to export its trace, log, and metric data in OpenTelemetry format. Hosted agents generate telemetry from the protocol runtime (the Responses or Invocations server) and from your agent code.

By default, the agent sends telemetry to the Application Insights resource that Foundry links to your project. You can also export the same data — using OpenTelemetry semantics — to any other OpenTelemetry-compliant endpoint or a self-hosted [OpenTelemetry Collector](https://opentelemetry.io/docs/collector/).

Exporting hosted agent telemetry via OpenTelemetry provides:

- Correlation of data across traces and logs from both the protocol runtime and agent code
- Consistent, standards-based telemetry generation
- Integration with any observability provider that consumes OpenTelemetry data

Key points to understand before configuring:

- Telemetry export is configured entirely through **environment variables** on the agent version — no instrumentation code changes are required. The protocol runtime autodetects the configuration at startup.
- Environment variables are set per version and are **immutable** once the version is created. To change the export destination, create a new agent version.
- Application Insights and an OTLP exporter can be active simultaneously. When both are configured, the agent sends telemetry to both destinations.

## Prerequisites

- A Foundry project with a deployed hosted agent — see [Deploy a hosted agent](/agent-service/deploy-hosted-agent)
- An OpenTelemetry-compliant endpoint if exporting to a provider other than Application Insights

## How hosted agents emit OpenTelemetry

Each hosted agent protocol (Responses, Invocations) uses a lightweight library that handles the HTTP or WebSocket server, health checks, and OpenTelemetry integration. These libraries bundle the OpenTelemetry distro, so the agent emits traces, logs, and metrics without any telemetry setup in code. The runtime selects its exporters based on the environment variables detected at startup:

- When the platform-injected `APPLICATIONINSIGHTS_CONNECTION_STRING` setting is present, the runtime exports to **Application Insights**. Foundry injects this setting automatically when project monitoring is enabled.
- When `OTEL_EXPORTER_OTLP_ENDPOINT` is present, the runtime also exports over **OTLP** to that endpoint.

Because both exporters use independent settings, telemetry can be sent to Application Insights, to an OTLP endpoint, or to both simultaneously.

<Note>
`APPLICATIONINSIGHTS_CONNECTION_STRING` is a platform-reserved setting that Foundry injects into the container — it cannot be set or overridden in `agent.yaml`. Foundry injects it only when monitoring is configured on the project. To stop sending telemetry to Application Insights, disable monitoring at the project level rather than trying to remove the setting from the agent.
</Note>

## Configure the export destination

### Application Insights

Application Insights is the default destination and requires no configuration on the agent. When project monitoring is enabled, Foundry links an Application Insights resource to the project and injects its connection string into every hosted agent container.

`APPLICATIONINSIGHTS_CONNECTION_STRING` is set automatically by Foundry. It is reserved and cannot be overridden in `agent.yaml`.

Telemetry appears in the linked Application Insights resource under **Investigate** > **Transaction search** or **Investigate** > **Performance**.

### OTLP exporter

Hosted agents support exporting to any OTLP-compliant endpoint. The required values depend on your observability provider.

**`OTEL_EXPORTER_OTLP_ENDPOINT`** — The OTLP endpoint URL provided by your observability backend. The endpoint URL is not a secret.

**`OTEL_EXPORTER_OTLP_PROTOCOL`** — The OTLP transport to use. Use `http/protobuf` (OTLP over HTTP) unless your provider requires otherwise — it's the most widely supported option. The exporter also supports `grpc` (OTLP over gRPC), but gRPC availability varies by provider. Match the protocol your endpoint expects.

**`OTEL_EXPORTER_OTLP_HEADERS`** — Headers for authentication credentials, such as an API key. The value is a comma-separated list of `key=value` pairs — for example, `api-key=YOUR_KEY`. Keep these format rules in mind:

- Provide the header name and format exactly as your provider documents them. A stray space or comma can cause silent authentication failures.
- For providers using HTTP Basic authentication, base64-encode the credentials yourself and pass `Authorization=Basic BASE64_CREDENTIALS`.
- URL-encode any value that contains reserved characters.
- To use different headers per signal, use the per-signal variants: `OTEL_EXPORTER_OTLP_TRACES_HEADERS`, `OTEL_EXPORTER_OTLP_LOGS_HEADERS`, and `OTEL_EXPORTER_OTLP_METRICS_HEADERS`.

Because the header value is a secret, store it in a connection rather than directly in `agent.yaml`. See [Store the OTLP authentication header in a connection](#store-the-otlp-authentication-header-in-a-connection).

Hosted agents support the standard environment variables defined in the [OpenTelemetry SDK configuration documentation](https://opentelemetry.io/docs/languages/sdk-configuration/). Consult your observability provider's documentation for required endpoint URLs, authentication methods, and any provider-specific settings.

## Set environment variables on your agent

Add the export settings to the `environment_variables` section of `agent.yaml`. When deploying with `azd`, these values are applied to the new agent version.

```yaml
environment_variables:
  - name: AZURE_AI_MODEL_DEPLOYMENT_NAME
    value: gpt-5-mini
  # Export OpenTelemetry traces, logs, and metrics over OTLP
  - name: OTEL_EXPORTER_OTLP_ENDPOINT
    value: https://YOUR_PROVIDER_OTLP_ENDPOINT
  - name: OTEL_EXPORTER_OTLP_PROTOCOL
    value: http/protobuf
```

This configuration sends data to *both* Application Insights and the OTLP endpoint. To send data only to the OTLP endpoint, disable monitoring at the project level.

<Tip>
If your infrastructure provisions the endpoint (for example, an OpenTelemetry Collector deployed alongside the agent), reference the endpoint with a deployment output — for example `value: ${OTEL_EXPORTER_OTLP_ENDPOINT}` — so Foundry resolves the value at deploy time instead of hard-coding it.
</Tip>

## Store the OTLP authentication header in a connection

The OTLP endpoint URL is not a secret, but the authentication header usually is. Keep the secret out of `agent.yaml` by storing it in a Foundry **Custom keys** project connection and referencing it with a placeholder.

1. Create a **Custom keys** (`CustomKeys`) connection on the project. In the Foundry portal, go to **Connected resources** > **Add connection** > **Custom keys**. Add a field that holds the full OTLP header string — for example, a field named `otlp_headers` set to `HEADER_NAME=SECRET_VALUE`. The exact header name and format come from your provider's documentation.

1. Reference the connection field from `agent.yaml` using a placeholder:

   ```yaml
   environment_variables:
     - name: OTEL_EXPORTER_OTLP_ENDPOINT
       value: https://YOUR_PROVIDER_OTLP_ENDPOINT
     - name: OTEL_EXPORTER_OTLP_PROTOCOL
       value: http/protobuf
     - name: OTEL_EXPORTER_OTLP_HEADERS
       value: ${{connections.otel-secrets.credentials.otlp_headers}}
   ```

   Foundry resolves the placeholder when the session sandbox starts and injects the value as a plain environment variable.

Keep these points in mind when using a connection:

- Create the connection **before** deploying the agent version. If the connection is missing when the sandbox starts, the placeholder resolves to an empty value.
- Secrets are write-only. A read of the connection returns `credentials: null`, so verify the value by reading the environment variable from inside the running container.
- The placeholder path must match the connection field name exactly.

## Send telemetry to both Application Insights and an OTLP endpoint

Because both exporters use independent settings, no extra configuration is needed to use both simultaneously. When both `APPLICATIONINSIGHTS_CONNECTION_STRING` (injected by the platform) and `OTEL_EXPORTER_OTLP_ENDPOINT` (set on the agent) are present, the agent sends telemetry to both destinations in parallel.

To send telemetry *only* to the OTLP endpoint, disable monitoring at the project level so Foundry doesn't inject the Application Insights connection string.

## Service name

Hosted agents report telemetry under a `service.name` fixed to the **agent's name** — shown as `cloud_RoleName` in Application Insights and as the service/resource name in the OTLP backend. The standard `OTEL_SERVICE_NAME` environment variable has no effect on a hosted agent; the platform overrides it. To use a different service name, name the agent accordingly.

## Tune exporter behavior

Adjust exporter behavior — such as trace sampling with `OTEL_TRACES_SAMPLER` — using standard OpenTelemetry environment variables. Trace sampling affects spans only; log records are exported regardless. The platform sets some values itself (for example, `service.name`), so resource attributes set through environment variables might be overridden. For the full list, see the [OpenTelemetry SDK configuration documentation](https://opentelemetry.io/docs/languages/sdk-configuration/).

## Considerations

- **Immutable versions.** Environment variables are set per agent version and cannot be changed after creation. To change the export destination or headers, create a new agent version.
- **Reserved Application Insights setting.** `APPLICATIONINSIGHTS_CONNECTION_STRING` cannot be set or overridden in `agent.yaml`. Control it through project monitoring.
- **Third-party data flow.** When exporting to a non-Microsoft observability provider, telemetry (which can include prompt, tool, and response content) flows to that provider. Review the provider's data handling, retention, and location policies, and confirm the flow meets your organization's compliance requirements.
- **Batched export.** Telemetry is exported in batches, so allow a few seconds after invoking the agent before data appears in your backend.

## Troubleshooting

### No telemetry appears at the OTLP endpoint

- Verify `OTEL_EXPORTER_OTLP_ENDPOINT` is set on the deployed agent version and points to the correct URL.
- Confirm `OTEL_EXPORTER_OTLP_PROTOCOL` matches what your endpoint expects (`http/protobuf` or `grpc`).
- Invoke the agent to generate telemetry, then allow a few seconds for the batched export.

### Authentication failures at the OTLP endpoint

- Confirm the `CustomKeys` connection exists and was created **before** the agent version was deployed.
- Verify the placeholder path in `agent.yaml` matches the connection field name exactly.
- Confirm the header string format matches your provider's documentation.
- Because secrets are write-only, verify the resolved value by reading `OTEL_EXPORTER_OTLP_HEADERS` from inside the running container.

### Telemetry still flows to Application Insights after configuring OTLP

Configuring an OTLP exporter doesn't disable Application Insights — both destinations receive telemetry when both are configured. To stop sending telemetry to Application Insights, disable monitoring at the project level.
